# Spec Write: 压成最小规格

把任务压缩成一份最小 spec。写清目标、范围、边界和验证方式。没有用户批准，不要展开具体实现。

## 触发条件

- 意图对齐通过后，准备开始具体实现
- 任务范围较大，需要拆分
- 用户说"写个 spec"、"压成最小"、"这轮只做什么"
- 防止模型把总目标偷渡进本轮

## 标准流程

1. 确认意图对齐已完成（总目标和边界已明确）
2. 把本轮任务压缩成 1-3 个具体动作
3. 输出 spec，等待用户批准
4. 写入到harness/spec.md , 写入前先检查是否存在这个文件
5. 用户批准后，进入执行阶段

## 输出格式

```
## 本轮 Spec

### 逻辑链条
- 上游依赖：[为什么要做这个？前置条件是什么？]
- 本轮动作：[具体做什么？]
- 下游影响：[做完之后会影响什么？解锁什么能力？]
- 目标对齐：[这轮动作如何服务于总目标？]

### 目标
[用一句话描述本轮要完成什么]

### 动作清单（1-3 个）
1. [具体动作 1：改哪个文件、改什么]
2. [具体动作 2：新增哪个组件]
3. [具体动作 3：删除哪段代码]

### 边界约束
- 不做：[明确列出不能碰的部分]
- 保持：[明确列出不能改的已有功能]

### 验证方式
- 构建：[具体命令]
- 测试：[具体命令]
- 手动：[具体检查方式]

等你批准后，我再动手。
```

## 规则

| 场景 | 做法 |
|------|------|
| 动作超过 3 个 | 拆成多轮，本轮只做最核心的 1-3 个 |
| 用户说"全做了" | 要提醒：先做最小可验证的一件事 |
| 不确定范围 | 问用户："XX 部分这轮做吗？" |
| 没有验证方式 | 必须补充，不能留空 |
| 逻辑链条断裂 | 检查：上游依赖是否存在？下游影响是否明确？和总目标是否对齐？ |

## 禁止

- 不要开始写代码
- 不要说"我先开始，边做边调整"
- 不要把"不做什么"留空
- 不要跳过验证方式

## 偏航信号

如果出现以下情况，说明 spec 写得有问题：

- 动作超过 3 个（任务太大，需要再拆）
- 没有"不做什么"（边界不清）
- 说"我先开始写，边写边调整"（逃避规划）
- 验证方式是"检查一下"（太模糊）
- 逻辑链条缺失：不知道为什么做、做完影响什么、和总目标什么关系
