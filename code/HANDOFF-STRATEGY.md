# HANDOFF 策略

> **作者**：艾森（Eisen）  
> **性质**：离线单文件 · 可自由分享 · 2026-06

---

## 什么是 HANDOFF 策略

让 AI 编程会话「无损续命」的交接纪律。适用于用 Claude Code / Codex / Cursor 做长周期、多会话项目时，在 `/clear`、重启、换机器之后，让新会话零摩擦接上全部进度。

**核心收益**：重启后第一条消息粘贴一段话，AI 立刻知道「做到哪了、证据是什么、下一步干什么、什么红线不能碰」——不靠 AI 的记忆功能，靠纪律。

---

## 01 · WHY：为什么需要它

两个真实痛点，任何重度使用 AI 编程工具的人都会撞上：

| 痛点 | 说明 |
|------|------|
| **① 上下文会丢** | 会话变长触发 compact（压缩摘要）后细节失真；`/clear` 或崩溃后直接归零。指望 AI「记得」是不可靠的。 |
| **② 进度会漂** | 更隐蔽的一条：实现不断前进，而计划文档停在三天前——长期不回写的变化就是不可见漂移。等你发现文档和现实对不上时，已经分不清哪个是真的了。 |

HANDOFF 策略针对的就是这两条：**用一个单一权威源文件对抗上下文丢失，用回写触发纪律对抗漂移。**

---

## 02 · ARCHITECTURE：机制总览——三层各司其职

| 层级 | 名称 | 说明 |
|------|------|------|
| **权威源** | `docs/<project>/HANDOFF-<日期>-<主题>.md` | 存全量状态，每个工作阶段一篇 |
| **路标** | `tasks/todo.md` 顶部「NEW SESSION 入口」 | 让任何新会话第一步就能找到最新一篇 |
| **点火器** | 重启开场白（一段可粘贴的话） | 把「读哪个文件、先验证什么、纪律红线」压缩成新会话的第一条消息 |

---

## 03 · STRUCTURE：HANDOFF 正文——最小六字段

每篇 HANDOFF 必须包含以下六段。多了可以，少了不行：

| 字段 | 内容 | 为什么必须有 |
|------|------|-------------|
| **① 一句话现状** | 当前阶段一句话说清 | 新会话 5 秒建立全局感 |
| **② git + 持久状态** | 分支、关键 commit hash、PR 状态、未 push 数量 | 代码状态是客观锚点，不会说谎 |
| **③ 本 session 成果** | 做了什么 + 证据（commit hash / test 结果 / 评审结论） | 没有证据的「完成」不算完成 |
| **④ backlog + 已关闭的决策** | 按 ROI 排序标理由；附「已否决项 + 一句话理由」负面清单 | 新会话不用重新想「接下来干嘛」，更不会重新打开你已经关掉的决策 |
| **⑤ 重启 verify 命令** | 第一条可直接跑的验证命令 + 不符时的含义 | 重启第一动作是核对现实，不是轻信文档 |
| **⑥ 环境约束与 gotcha** | 新会话不知道就会浪费 ≥10 分钟或造成损害的非显而易见事实（必须的后台服务、必带的环境变量、危险操作的正确姿势） | 工作状态会过期，环境约束会咬人 |

### 第 ⑤ 条的精髓

新会话开工前先跑这几条命令（看 git log 顶端 hash、跑一个测试、查一个状态文件），确认现实与 HANDOFF 描述一致，再开始干活。文档可能过时，命令输出不会。

每条命令除了「预期输出 X」，再补一句「不符 → 意味着什么」——这让 verify 从核对升级为诊断。

### 第 ④ 条的负面清单

AI 恢复会话后最容易犯的错不是「不知道做什么」，而是重新提议你已经否决过的方案。「Y 已否决（理由 Z），别再试」一句话，省掉一轮完整的重复论证。

## 04 · DISCIPLINE：回写纪律——对抗漂移的关键

### 回写触发与 "Done" 门槛

| 回写触发 | "Done" 门槛 |
|----------|-------------|
| 每完成一个带验证的工作单元（一次 commit + 测试绿 / 一个决策落定 / 一个阻塞解除），立刻把「进度 + 证据 + 剩余未决」投影回当前 HANDOFF。不要攒到 session 末——攒着就是在累积漂移，而且 session 可能意外死掉。 | 仅当 artifact / decision / verification 真实存在时才标完成。测试没跑就不写「测试通过」；PR 没合就不写「已合并」。HANDOFF 的全部价值建立在它不说谎上。 |

### 四大纪律原则

| 原则 | 说明 |
|------|------|
| **历史不回溯改** | HANDOFF 是历史文档。事实变了就写一篇新的（frontmatter 用 `supersedes` 链起来），不要回去改旧的——改了就失去了「当时的真实状态」这条审计线索。 |
| **单一权威源** | 不要为同一信息维护平行的第二套「规划树 / 状态板」。两套系统必然分叉，重启时你要先花十分钟判断「哪棵树是活的」——这比没有文档更糟。 |
| **篇内过时走勘误层** | 「不回溯改」留下一个缝：HANDOFF 写完三小时后某段就过时了（如「分支待 merge」→ 已 merge），改旧篇违纪、开新篇又太重。解法：让 todo 入口承担对最新一篇的勘误层——「其中『X』段已过时，以本入口为准」。HANDOFF 是快照，入口是增量勘误，合起来才是当前真相。 |
| **HANDOFF 是状态，不是知识库** | 它只保「接下来要用的状态」。踩坑教训、设计决策、可复用经验必须升华到别处（`LESSONS.md` / `solutions` 目录 / 记忆系统）——否则三十篇之后要靠考古找知识。判据：这条信息下个 session 之后还有用吗？有 → 不属于 HANDOFF。 |

## 05 · ENTRY POINT：todo.md 入口的维护

`tasks/todo.md` 顶部维护一个「🚀 NEW SESSION 入口」段：

| 要素 | 说明 |
|------|------|
| **先读** | 指向最新一篇 HANDOFF；HANDOFF 某段已过时的，在这里标注「其中『X』段已过时——以本入口为准」（勘误层） |
| **盘面** | 3–6 条 bullet 浓缩当前状态，含红线提示（如「某操作绝不自动执行」） |
| **历史入口** | 旧入口降级不删除：标上【已完成：见上方新入口】沉到下面，保留演进轨迹 |

每次 HANDOFF 更新后顺手刷新这个入口——它是新会话的第一落点。入口保持指针 + 勘误 + 红线的轻量形态，状态细节留在 HANDOFF 里，别让入口长成第二份 HANDOFF。

## 06 · TEMPLATE：重启开场白模板

新会话第一条消息直接粘这一段（每次交接时由 AI 提前帮你拟好）。

红线必须写成可判定指令——「绝不自动执行 X」「push 等我指令」这种 AI 能严格执行的措辞；「小心点」对 AI 等于没写。如果你没有全局配置文件兜底，开场白就是唯一的纪律载体，措辞强度决定执行强度。

### 模板格式

```
项目 <路径>（<项目名>）。读 docs/<project>/HANDOFF-<最新>.md + tasks/todo.md 顶部入口。
上个 session：<一句话成果，含关键 commit hash>。
⚠ <当前环境的特殊状态，如有>
现在做：① <下一步1> ② <下一步2>。
纪律不变：<红线1>；<红线2>。
重启 verify：<可直接复制运行的命令串>（预期输出 <X>）。
```

### 真实例子（脱敏）

```
项目 ~/projects/memory（Anamnesis）。读 docs/anamnesis/HANDOFF-2026-06-10-xxx.md + tasks/todo.md 顶部入口。
上个 session：5 项借鉴全完成——#2 验证、#6/#5/#10 落 main、#1 在 feat/xxx 分支（8 commit，评审全过，1158 test 0 fail）。
⚠ 当前 checkout 在该分支 = 生产 hook 正在试用新代码，切回 main 会回退。
现在做：① review/merge 两个分支——由我拍板，不自动 merge；② 继续观察期。
纪律不变：高危操作绝不自动执行；push 等我指令。
重启 verify：git log --oneline -2（预期 749132f/c30d588）+ sqlite3 查两表行数相等。
```

## 07 · TEMPLATE：HANDOFF 文档模板

```markdown
---
title: HANDOFF <日期> — <主题一句话>
date: YYYY-MM-DD
status: <一句话定性，如「全部实施完毕，X 待用户拍板」>
supersedes: <被取代的上一篇文件名，可选>
---

# ① 一句话现状
<当前阶段在干什么、到哪了>

# ② git + 持久状态
- main：<commit hash 列表 + 是否已 push>
- <分支名>（N commit）：<hash 串> —— <状态>
- ⚠ <环境特殊状态，如「生产正在跑分支代码」>

# ③ 本 session 成果
1. <成果> —— 证据：<commit / test 结果 / 评审结论>

# ④ 下一步 backlog（按 ROI）
1. <最高优先> —— <理由>

已关闭的决策（别再提议）：
- <方案 Y> 已否决 —— <一句话理由>

# ⑤ 重启 verify
```bash
<命令1> # 预期输出 X；不符 → <意味着什么>
<命令2> # 预期输出 Y；不符 → <意味着什么>
```

# ⑥ 环境约束与 gotcha
- <必须在跑的后台服务 / 必带的环境变量>
- <危险操作的正确姿势，如「复制数据库必须用 X 方式，直接 cp 会丢数据」>
```

## 08 · PITFALLS：常见坑

| 坑 | 后果 | 解法 |
|----|------|------|
| 攒到 session 末才写 | session 崩溃 = 全丢；写时已记不清细节 | 每个验证过的工作单元立即回写 |
| 写「应该完成了」 | 新会话基于谎言开工，越走越偏 | 没证据不标 Done |
| 同时维护两套状态文档 | 重启先考古哪套是真的 | 单一权威源，另一套删掉 |
| verify 命令缺失或不可跑 | 新会话盲信文档 | 写下你自己重启时真的会跑的那几条 |
| 回去改旧 HANDOFF | 丢失历史真实状态 | 新事实写新篇 + supersedes；篇内小过时走入口勘误层 |
| 入口不更新 | 新会话读到三周前的 HANDOFF | 每篇新 HANDOFF 落盘时顺手刷 todo 入口 |
| 把教训/经验埋进 HANDOFF | 知识沉入只写不读的日志链，要用时靠考古 | 耐久知识升华到 LESSONS.md / solutions 目录，HANDOFF 只留状态 |
| backlog 只写要做的，不写否决的 | AI 重新提议你已经关掉的方案，重复论证一轮 | 字段 ④ 附「已关闭的决策」负面清单 |

## 起步建议：不用一次全上

**最小可行版本只要四步**：

1. 今天 session 结束前，让 AI 按「最小六字段」写第一篇 HANDOFF
2. 让 AI 顺手生成一段重启开场白，存在 `todo.md` 顶部
3. 下次新会话粘贴开场白，体验一次「无损接上」
4. 之后再建立「工作单元级回写」的习惯——这是从「有用」到「可靠」的分界线

## 理念出处

「不可见漂移」概念借鉴自 plan-tree 项目（评估后采纳理念、未引入工具——理念比工具便宜）。整套纪律在一个 100+ 会话的真实项目上驯化了一个月后稳定成型。
